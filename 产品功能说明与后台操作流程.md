# 小原圈产品功能说明与后台运营流程

> 产品名：小原圈  
> 定位：面向二手衣物/小用品交易的社区化交易平台，包含商品交易、社区互动、认证信任、IM、支付/钱包、售后、举报与后台审核运营能力。  
> 本文用于上线测试、客户注册认证演示、后台运营培训与验收。

## 1. 产品角色

### 1.1 客户/普通用户

客户可以作为买家、卖家或社区用户使用平台：

- 注册/登录账号。
- 浏览商品、收藏商品、搜索商品、查看卖家主页。
- 发布二手商品，管理自己的衣橱/商品。
- 下单、支付、查看订单、确认收货、评价。
- 发起售后申请。
- 绑定提现账号，查看钱包流水，发起提现。
- 发送聊天消息。
- 发布社区内容、评论、点赞、关注。
- 提交视频认证，成为可信卖家。
- 举报商品、用户或聊天内容。

### 1.2 后台管理员/运营人员

管理员通过独立管理后台进行运营：

- 查看运营首页数据。
- 处理认证、举报、商品、提现等审核。
- 查询用户与用户详情。
- 查询订单与交易状态。
- 处理售后申请。
- 复核提现申请。
- 配置定位服务。
- 查看后台操作日志。
- 由具备 `operator:grant` 的最高授权管理员进入“运营授权”，给运营经理分配审核、财务、用户、订单、售后、系统配置、审计日志等权限。

### 1.3 系统服务

系统负责：

- 注册/登录与 token/session 管理。
- 媒体上传票据签发。
- 商品、订单、支付、钱包、售后等状态流转。
- 后台权限校验与操作审计。
- 敏感信息脱敏与安全拦截。

## 2. 用户端功能说明

## 2.1 注册/登录

入口：用户端登录页。

当前后端接口：

```text
POST /api/auth/login
```

请求字段：

- `mobile`：手机号。
- `password`：密码。

业务规则：

1. 新手机号首次登录会自动创建 `user_account` 与 `user_profile`。
2. 已存在手机号使用密码登录。
3. 登录成功后返回 `accessToken` 与 `refreshToken`。
4. 用户端后续请求通过 `Authorization: Bearer <accessToken>` 访问。
5. 密码不应明文存储，后端使用哈希校验。

上线测试验收点：

- 新手机号能完成注册/登录。
- 登录后刷新页面仍能保持 token。
- 未登录访问敏感操作时应提示登录。
- 弱密码、非法手机号应被拒绝。

## 2.2 个人资料与账号安全

相关页面：

- 我的。
- 个人资料编辑。
- 账号安全。
- 收货地址。

核心能力：

- 查看当前用户资料。
- 编辑昵称、城市、简介等资料。
- 管理收货地址，设置默认地址。
- 查看账号安全信息与最近登录信息。

验收点：

- 资料修改后刷新仍保持服务端数据。
- 收货地址创建、默认设置、删除均落库。
- 页面不能展示敏感 token、密码或未脱敏账号。

## 2.3 视频认证/卖家认证

用户端入口：认证页面。

提交接口：

```text
POST /api/audit/video-identity
```

媒体票据要求：

- 视频认证素材必须使用 `VIDEO_IDENTITY` 上传票据。
- 用户端不能把本地预览地址、占位 URL、未签发外链直接提交保存。

客户流程：

1. 客户登录。
2. 进入认证页面。
3. 上传视频认证材料。
4. 后端签发 `VIDEO_IDENTITY` 媒体票据。
5. 客户提交认证说明与视频 URL。
6. 系统生成待审核记录。
7. 管理员在后台审核通过或驳回。
8. 审核通过后，用户资料中的视频认证状态变为已通过。
9. 只有后端持久化状态为已通过的卖家，公开主页才展示“视频认证卖家”信任卡。

后台处理：见本文第 4.3 节。

验收点：

- 未登录不能提交认证。
- 认证提交后后台能看到待审核记录。
- 通过后公开卖家主页展示信任卡。
- 待审核、驳回、默认状态不能展示信任卡。
- 驳回原因或备注应可被审计追踪。

## 2.4 商品浏览、搜索与收藏

相关页面：

- 首页。
- 分类。
- 搜索结果。
- 商品详情。
- 收藏。
- 卖家公开主页。

核心接口：

```text
GET /api/products
GET /api/products/{productId}
GET /api/products/seller/{sellerId}
GET /api/products/favorites
POST /api/products/{productId}/favorite
DELETE /api/products/{productId}/favorite
GET /api/user/{userId}/profile
```

业务规则：

- 首页、分类、搜索必须使用真实商品数据。
- 商品详情必须展示真实卖家信息。
- 收藏状态必须落库，不能只在前端临时切换。
- 卖家主页认证信息必须以后端认证状态为准。

验收点：

- 列表、详情、收藏刷新后状态一致。
- 搜索结果不使用固定假数据。
- 商品图片来自合规上传票据或可信资源。

## 2.5 商品发布与编辑

相关页面：

- 发布。
- 商品编辑。
- 我的衣橱。

核心接口：

```text
POST /api/products
PUT /api/products/{productId}
PUT /api/products/{productId}/visibility
GET /api/products/mine
```

媒体票据要求：

- 商品图片必须使用 `PRODUCT_IMAGE` 上传票据。

建议业务流：

1. 卖家登录。
2. 填写标题、描述、价格、分类、交易规则。
3. 上传商品图片。
4. 后端创建商品，初始状态进入待审核或不可售状态。
5. 后台审核商品。
6. 审核通过后商品进入可售列表。
7. 卖家可编辑商品或调整上下架可见性。

验收点：

- 发布商品后后台可审核。
- 未审核商品不应作为已通过商品展示给普通买家。
- 编辑商品后详情页数据同步。

## 2.6 订单与支付

相关页面：

- 确认订单。
- 支付方式。
- 支付收银台。
- 订单列表。
- 订单详情。
- 发货。
- 物流。
- 评价。

核心接口：

```text
POST /api/orders
GET /api/orders
GET /api/orders/{orderNo}
POST /api/orders/{orderNo}/pay
POST /api/orders/{orderNo}/ship
POST /api/orders/{orderNo}/confirm-receipt
POST /api/orders/{orderNo}/review
```

业务规则：

1. 买家从商品详情发起下单。
2. 系统生成订单号。
3. 支付必须绑定真实订单号。
4. 支付成功后订单状态进入待发货或已支付。
5. 卖家发货。
6. 买家确认收货。
7. 交易完成后可评价。

上线测试注意：

- 测试环境可保留沙箱支付或受控模拟支付，但必须标记清楚，不得把前端按钮点击当作支付成功依据。
- 支付回调、订单金额、订单号、支付流水必须以后端记录为准。
- `simulate-success` 只能用于开发验证，不应作为正式上线测试结论。

验收点：

- 支付动作不能绕过订单号。
- 买家、卖家、管理员看到的订单状态一致。
- 非订单买家不能确认收货或申请售后。

## 2.7 售后

相关页面：

- 售后申请。
- 售后详情。

核心接口：

```text
POST /api/after-sales
GET /api/after-sales/{afterSalesNo}
```

媒体票据要求：

- 售后凭证必须使用 `AFTER_SALES_EVIDENCE` 上传票据。

业务规则：

1. 只有已支付订单买家可以发起售后。
2. 退款金额必须大于 0，且不能超过订单金额。
3. 售后申请写入 `after_sales_record`。
4. 管理员后台审核通过或驳回。
5. 审核结果同步售后详情。

验收点：

- 非买家不能发起该订单售后。
- 超额退款被拒绝。
- 凭证 URL 必须来自后端票据。
- 后台审核后状态持久化。

## 3. 管理后台功能说明

## 3.1 后台登录与会话

入口：管理后台 `/login`。

接口：

```text
POST /api/admin/session/login
GET /api/admin/session/me
POST /api/admin/session/logout
```

登录成功返回：

- `username`
- `userId`
- `permissions`
- `sessionId`
- `expiresAt`

后台受保护接口请求头：

```text
X-User-Id: <管理员用户ID>
X-Admin-Session: <adm_开头的sessionId>
```

权限规则：

- `audit:read`：读取审核。
- `audit:review`：处理认证、举报、商品审核。
- `finance:read`：读取提现。
- `finance:review`：复核提现。
- `user:read`：查询用户。
- `order:read`：查询订单。
- `after-sales:read`：读取售后。
- `after-sales:review`：审核售后。
- `system:config`：系统配置。
- `audit:log`：查看操作日志。

验收点：

- 未登录访问后台页面跳转登录。
- 无权限访问对应菜单/接口应被拒绝。
- 退出后 session 失效。
- session 过期后需重新登录。

## 3.2 运营首页

页面：`/dashboard`

接口：

```text
GET /api/admin/dashboard
```

展示内容：

- 平台关键指标汇总。

操作要求：

## 3.3 认证/举报/商品/提现审核

页面：

- `/audit`
- `/audit/:auditNo`

接口：

```text
GET /api/admin/audit
GET /api/admin/audit/{auditNo}
POST /api/admin/audit/{auditNo}/approve
POST /api/admin/audit/{auditNo}/reject
POST /api/admin/products/{productId}/approve
```

审核对象：

- 视频认证。
- 用户举报。
- 商品发布审核。
- 提现复核关联审核。

认证审核流程：

1. 管理员进入审核列表。
2. 筛选/查看 `VIDEO_IDENTITY` 类型记录。
3. 打开详情，核对用户、认证视频、描述、历史记录。
4. 通过：用户 `video_identity_status` 变为通过，公开主页可展示“视频认证卖家”。
5. 驳回：认证状态变为驳回，公开主页不能展示认证信任卡。
6. 系统记录管理员操作日志。

举报处理流程：

1. 管理员查看举报目标与凭证。
2. 判断是否违规。
3. 通过/处理：记录处理结果，必要时下架商品或限制用户。
4. 驳回：标记举报不成立并记录备注。

商品审核流程：

1. 查看待审核商品。
2. 检查标题、图片、价格、描述、是否违规。
3. 点击通过后商品进入可售状态。
4. 违规商品不应上架。

提现审核流程：

1. 查看提现申请与提现账号脱敏信息。
2. 核对钱包余额、提现金额、用户身份与风险记录。
3. 通过或驳回。
4. 记录财务复核日志。

验收点：

- 每次审核都应写入后台操作日志。
- 审核备注不能泄露完整银行卡、密码、token。
- 视频认证通过前，前台不能显示认证卖家信任卡。

## 4. 上线测试核心业务流程

## 4.1 客户注册/登录流程

1. 打开用户端 H5。
2. 进入登录页。
3. 输入新手机号和符合规则的密码。
4. 调用 `POST /api/auth/login`。
5. 后端创建账号和资料。
6. 用户端保存 `accessToken`。
7. 进入“我的”页面查看用户资料。
8. 刷新页面，确认仍保持登录态。

验收结果：

- 数据库有 `user_account` 记录。
- 数据库有 `user_profile` 记录。
- 用户端所有需登录接口不再提示未登录。

## 4.2 客户认证流程

1. 客户登录。
2. 进入“我的/认证”。
3. 上传视频认证素材。
4. 后端通过 `POST /api/media/upload-tickets` 签发 `VIDEO_IDENTITY` 票据。
5. 用户提交认证。
6. 后端生成审核记录。
7. 管理员后台登录。
8. 进入 `/audit` 查看认证记录。
9. 打开详情。
10. 管理员通过或驳回。
11. 客户刷新个人资料/公开主页查看状态变化。

通过标准：

- 通过后公开视频认证卖家信任卡出现。
- 驳回/待审核时不显示信任卡。
- 后台日志有审核记录。

## 4.3 商品发布与后台审核流程

1. 卖家登录。
2. 进入发布页。
3. 上传 `PRODUCT_IMAGE` 商品图片。
4. 填写商品标题、价格、描述、分类。
5. 提交商品。
6. 后台运营进入审核列表或商品审核入口。
7. 审核通过。
8. 买家在首页/分类/搜索看到商品。
9. 买家进入商品详情并可下单。

通过标准：

- 商品审核前不作为已通过商品展示。
- 商品审核后列表和详情可见。
- 审核操作有日志。

## 4.4 下单支付流程

1. 买家登录。
2. 选择已审核商品。
3. 进入确认订单。
4. 提交订单。
5. 系统生成订单号。
6. 买家进入支付页。
7. 完成沙箱支付或测试支付。
8. 订单进入已支付状态。
9. 卖家发货。
10. 买家确认收货。
11. 买家评价。

通过标准：

- 订单号贯穿确认订单、支付、订单详情。
- 支付结果以后端状态为准。
- 后台订单列表可看到订单状态变化。

## 4.5 售后流程

1. 买家对已支付订单发起售后。
2. 上传 `AFTER_SALES_EVIDENCE` 凭证。
3. 填写退款金额和原因。
4. 后端生成售后编号。
5. 管理员进入 `/after-sales`。
6. 打开详情，核对凭证与金额。
7. 审核通过或驳回。
8. 买家查看售后结果。

通过标准：

- 非订单买家无法申请。
- 金额超限无法提交。
- 后台审核状态可回传到用户端。

## 4.6 提现流程

1. 卖家完成交易并产生可提现余额。
2. 进入钱包。
3. 绑定提现账号。
4. 发起提现申请。
5. 后台财务进入 `/finance/withdrawals`。
6. 核对金额和账号脱敏信息。
7. 通过或驳回。
8. 钱包流水记录提现状态。

通过标准：

- 提现账号脱敏。
- 提现状态前后台一致。
- 财务复核写入日志。

## 4.7 举报处理流程

1. 用户在商品/用户/聊天/社区内容中发起举报。
2. 上传 `REPORT_EVIDENCE` 凭证。
3. 填写原因和描述。
4. 后台进入 `/audit` 查看举报。
5. 管理员处理举报。
6. 系统记录处理结果。

通过标准：

- 举报目标不能为空。
- 后台可查看目标和凭证。
- 处理记录脱敏并可追踪。

## 4.8 运营经理授权流程

1. 最高授权管理员完成后台真实登录。
2. 登录会话必须包含 `operator:grant` 权限。
3. 进入管理后台 `/operators`（菜单名：运营授权）。
4. 输入要授权的运营经理用户 ID，点击加载权限。
5. 后台调用 `GET /api/admin/operators/{userId}/permissions`，只返回用户编号、昵称、状态和已启用权限码。
6. 勾选需要授予的权限：`audit:read`、`audit:review`、`finance:read`、`finance:review`、`user:read`、`user:risk-control`、`order:read`、`after-sales:read`、`after-sales:review`、`system:config`、`audit:log`、`operator:grant`。
7. 点击保存授权，后台调用 `POST /api/admin/operators/{userId}/permissions`。
8. 后端禁用该用户旧权限并写入新权限，同时写入 `OPERATOR_PERMISSION_GRANT` 后台审计日志。
9. 运营经理重新登录或刷新会话后，只能看到自身权限允许的后台菜单与页面。

通过标准：

- 无 `operator:grant` 的后台账号访问 `/operators` 或接口时必须被拒绝。
- 未知权限码、空权限列表、停用用户必须被后端拒绝，不能产生授权变更。
- 授权页面不使用本地样例数据，加载或保存失败时只显示失败提示。
- 授权成功后 `admin_user_permission` 持久化更新，并能在审计日志中追踪操作者、目标用户和权限摘要。

## 6. 后台操作培训简版

### 6.1 每天运营先看什么

1. 登录管理后台。
2. 看运营首页待处理卡片。
3. 优先处理认证、商品、售后、提现、举报。
4. 查看异常订单和高风险用户。
5. 下班前检查操作日志是否有异常失败记录。

### 6.2 认证审核怎么判

通过：

- 视频能证明用户真实性。
- 内容无违规。
- 素材来源为合规上传票据。

驳回：

- 视频无法打开。
- 视频与账号信息明显不符。
- 使用外链/占位/非票据素材。
- 存在违规内容。

### 6.3 商品审核怎么判

通过：

- 商品图片清晰。
- 标题、描述、价格合理。
- 属于平台允许交易范围。

驳回/不通过：

- 图片违规或疑似盗图。
- 文案包含引流、诈骗、敏感信息。
- 商品不属于允许类目。
- 价格明显异常且无法解释。

### 6.4 售后审核怎么判

通过：

- 买家为订单真实买家。
- 凭证清晰有效。
- 退款金额不超过订单金额。
- 订单状态支持售后。

驳回：

- 凭证无效。
- 非订单买家申请。
- 金额超限。
- 已超过平台售后规则。

### 6.5 提现审核怎么判

通过：

- 用户余额充足。
- 提现账号已验证。
- 金额符合规则。
- 无明显风控风险。

驳回：

- 余额不足。
- 账号异常。
- 用户存在争议订单或风控记录。
- 金额异常。

## 7. 当前上线测试重点

当前项目已具备用户端、后端、管理后台的主要页面和接口骨架。上线测试阶段优先补全和验证：

1. 真实客户注册登录到认证审核闭环。
2. 后台真实账号登录与权限验证。
3. 商品发布审核到下单支付闭环。
4. 售后、提现、举报三条运营处理链路。
5. 媒体上传票据与敏感信息脱敏。
6. 生产构建关闭 mock/dev header。

不建议当前阶段过度投入：

- 像素级 UI 微调。
- 动效细节。
- 非核心营销页面。
- 深度性能压测。

先保证“客户能注册、能认证、能发布、能交易、后台能审、财务能复核、售后能处理、日志可追踪”。
