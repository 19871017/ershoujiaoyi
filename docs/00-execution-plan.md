# 总执行计划：二手闲置社交交易平台

版本：V1.0
状态：已冻结 MVP 范围与总体架构
用途：作为项目落地阶段的统一执行基线，供主任务拆分、子 agent 分工、接口联调、测试验收使用。

## 1. 冻结目标

本文件冻结以下内容，作为当前阶段的唯一执行口径：

- 冻结 MVP 范围，不再新增新业务线。
- 冻结核心服务边界，避免跨服务反复拆改。
- 冻结第一批实现顺序，先打通主交易闭环，再扩展周边能力。
- 冻结不可并行修改区域，减少多团队互相阻塞。
- 冻结阶段交付物，按阶段验收，不以“顺手补功能”的方式扩范围。

## 2. MVP 范围

### 2.1 MVP 的目标

MVP 目标不是做完整平台，而是验证以下最小闭环是否成立：

1. 用户可注册、登录、完成基础身份选择。
2. 卖家可走人工认证，认证后可发布闲置商品。
3. 买家可完成自动认证，具备基础购买、聊天、关注和收礼能力。
4. 商品可浏览、确认交易准则、下单、支付、生成订单。
5. 平台可完成基础 IM 聊天、基础礼物、基础钱包账本。
6. 平台可完成基础内容审核和后台审核处理。
7. 平台可展示基础榜单入口与基础榜单数据。

### 2.2 MVP 必做范围

MVP 只包含以下内容：

- 登录 / 注册 / 退出登录
- 身份选择：买家、卖家
- 卖家人工认证申请、审核、通过 / 驳回
- 买家自动认证规则落地
- 用户基础资料页
- 闲置商品发布：仅图片或多图，不支持视频
- 商品列表、商品详情、类目基础浏览
- 交易准则展示、勾选确认、订单快照保存
- 下单、支付、订单创建、订单状态流转
- 基础 IM 聊天：文本消息、基础图片消息、会话列表
- 关注、互关、好友关系
- 礼物基础版：礼物列表、赠送、收礼记录、基础展示
- 钱包账本基础版：充值、消费、收支流水、余额、冻结记录
- 女神榜 / 男神榜基础版：按既定规则汇总展示
- 内容审核基础版：商品、头像、昵称、聊天消息的基础审核与人工复审入口
- 后台基础能力：认证审核、商品审核、提现审核、基础配置管理

### 2.3 MVP 明确不做范围

以下内容不进入 MVP，禁止在第一阶段插入：

- 论坛 / 公共交流区完整功能
- 发货凭证完整链路
- 售后纠纷系统
- 高级风控规则引擎
- 地区等级搜索权限完整开放
- 视频认证全自动化
- 私聊视频能力完整审核链路
- 礼物异常检测规则系统
- 数据看板与经营分析
- 推荐算法
- 用户风险评分体系
- 盗图检测
- 大模型辅助审核
- 复杂运营活动、新手任务、成长权益

### 2.4 MVP 验收口径

MVP 以“主闭环可跑通”为验收标准，不以功能数量为标准。验收必须覆盖：

- 注册、登录、身份选择可用
- 卖家认证审核可用
- 商品发布到下单支付可用
- 聊天、关注、礼物、钱包可用
- 审核与后台处理可用
- 订单、账本、审核记录均可追溯

## 3. 服务边界

### 3.1 总体服务拆分原则

服务边界按“业务高内聚、数据强归属、接口清晰、低耦合”划分。MVP 阶段优先保证边界清楚，不追求过度微服务化。

### 3.2 建议服务边界

#### 3.2.1 auth-service
职责：

- 注册、登录、退出、验证码、Token 管理
- 登录态校验、基础风控校验
- 第三方登录预留

边界：

- 不管理用户资料正文
- 不管理身份标签
- 不直接写商品、订单、钱包业务数据

#### 3.2.2 user-service
职责：

- 用户资料、头像、昵称、简介
- 身份状态：买家 / 卖家
- 卖家认证状态、买家自动认证状态
- IP 属地、地区展示基础信息
- 关注 / 互关 / 好友关系

边界：

- 不承载支付逻辑
- 不承载订单状态机
- 不承载内容审核判断本身

#### 3.2.3 goods-service
职责：

- 闲置商品发布、编辑、上下架
- 商品图片、类目、价格、描述、交易准则
- 商品详情、列表、搜索基础能力

边界：

- 不负责支付扣款
- 不负责订单流转
- 不负责审核策略决策，只接收审核结果

#### 3.2.4 order-service
职责：

- 下单、取消、支付确认、订单状态流转
- 订单快照：商品信息、交易准则、费率快照
- 订单与钱包、支付服务的状态协调

边界：

- 不直接处理充值账户逻辑
- 不直接处理商品编辑逻辑
- 不直接承担 IM 消息存储

#### 3.2.5 payment-service
职责：

- 充值、支付、退款、支付结果回调
- 支付渠道封装
- 支付流水与外部渠道对账接口

边界：

- 不承担订单状态的主控权
- 不承担钱包总账归集
- 不承载业务规则审批

#### 3.2.6 wallet-service
职责：

- 钱包余额、冻结、解冻、流水、账本
- 礼物收入、商品收入、提现前冻结
- 财务可追溯记录

边界：

- 不决定支付渠道
- 不决定订单是否成交
- 不承担内容审核

#### 3.2.7 chat-service
职责：

- 单聊会话、消息收发、会话列表
- 基础文本 / 图片消息
- 消息送达状态、基础审核接入

边界：

- 不负责好友关系计算
- 不负责卖家认证
- 不负责礼物账务

#### 3.2.8 gift-service
职责：

- 礼物配置、赠送、收礼记录
- 礼物费率快照、收礼展示数据
- 礼物收益事件输出给钱包

边界：

- 不直接修改钱包总余额
- 不决定用户认证
- 不承担 IM 存储

#### 3.2.9 audit-service
职责：

- 商品、头像、昵称、聊天内容的审核接入
- 自动审核结果、人工复审、违规留痕
- 审核任务、审核记录、证据快照

边界：

- 不直接改写业务主数据，只回传审核结果
- 不负责处罚执行，由各业务服务落地状态

#### 3.2.10 admin-service
职责：

- 后台登录与权限
- 认证审核、商品审核、提现审核
- 基础配置：手续费、分成、礼物配置
- 审核结果流转

边界：

- 不直接暴露业务核心写库接口给前台
- 不绕过业务服务直接修改主表

### 3.3 服务间依赖关系

推荐依赖顺序：

- auth-service 提供统一登录入口
- user-service 依赖 auth-service 的身份信息
- goods-service 依赖 user-service 的卖家认证状态
- order-service 依赖 goods-service、payment-service、wallet-service
- gift-service 依赖 user-service、wallet-service
- chat-service 依赖 user-service、audit-service
- audit-service 为横切服务，向各业务服务输出审核结果
- admin-service 调度各业务服务的审核与配置能力

### 3.4 边界冻结原则

以下边界在 MVP 周期内不允许随意打散：

- 认证状态归 user-service
- 商品主数据归 goods-service
- 订单状态归 order-service
- 钱包账本归 wallet-service
- 礼物配置与礼物交易归 gift-service
- 审核任务与审核记录归 audit-service
- 后台操作归 admin-service

## 4. 阶段划分

### 阶段 0：架构冻结与底座建立
目标：统一代码结构、接口规范、数据规范、权限模型、日志规范。

交付物：

- 项目目录结构
- 服务边界说明
- API 约定
- 数据库基础规范
- 环境配置与部署方式
- 通用返回码、异常码、日志规范

验收标准：

- 各子服务可独立启动
- 基础鉴权与公共组件可复用
- 统一规范文档已固化

### 阶段 1：MVP 主闭环
目标：跑通“注册登录 → 身份认证 → 发商品 → 看商品 → 下单支付 → 聊天 / 礼物 / 钱包 → 后台审核”的最小链路。

交付物：

- 前台核心页面
- 核心接口
- 基础账本
- 基础审核
- 后台审核台

验收标准：

- 卖家和买家的核心动作均可跑通
- 所有资金动作可追踪
- 审核结果可落库、可追溯

### 阶段 2：稳定运营增强
目标：补齐交易安全、纠纷、地区权限、发货凭证、数据运营能力。

交付物：

- 发货凭证
- 售后纠纷
- 地区等级搜索权限
- 论坛基础能力
- 风控规则
- 数据看板

### 阶段 3：增长优化
目标：补齐运营增长工具与智能化能力。

交付物：

- 推荐算法
- 风险评分
- 盗图检测
- 大模型辅助审核
- 运营活动与任务体系

## 5. 并行原则

### 5.1 可以并行推进的区域

在不改动同一主数据表和同一状态机前提下，可并行推进：

- 前端页面开发与后端接口开发并行
- 商品模块与 IM 模块并行
- 礼物模块与榜单模块并行
- 后台审核页与前台个人中心并行
- 审核服务与钱包服务并行
- 接口文档整理与数据库脚本准备并行

### 5.2 并行的前提条件

所有并行必须满足：

- 已冻结字段名、接口名、事件名
- 已确定主数据归属服务
- 已明确状态机流转方向
- 已约定统一错误码和回调约束
- 已确定联调 Mock 数据格式

### 5.3 并行执行要求

- 每个子 agent 只改自己负责的服务或页面目录
- 跨服务改动必须先确认接口契约
- 先写契约，后写实现
- 先建主流程，再补边角流程
- 先单测，再联调，再集成

## 6. 不可并行修改区域

以下区域属于强耦合，禁止多个子 agent 同时修改同一套定义：

### 6.1 不可并行的核心定义区

- 数据库核心表结构
- 用户身份与认证状态定义
- 订单状态机
- 钱包账本与流水定义
- 费率快照与分成规则
- 礼物订单结构
- 审核任务与审核结果结构
- 权限角色模型

### 6.2 不可并行的接口契约区

- 登录 / Token 约定
- 商品发布与下单接口
- 支付回调接口
- 钱包入账 / 出账接口
- 审核回调接口
- IM 消息入库格式
- 礼物赠送接口

### 6.3 不可并行的状态机区

以下状态机必须由单一责任人先冻结，再允许其他模块调用：

- 用户认证状态机
- 商品上下架状态机
- 订单状态机
- 支付状态机
- 钱包冻结 / 解冻状态机
- 审核流转状态机
- 礼物赠送与入账状态机

### 6.4 不可并行的原因

这些区域一旦多方同时修改，最容易出现：

- 字段命名不一致
- 状态流转冲突
- 账务重复记账
- 回调幂等缺失
- 前后端契约不一致
- 联调期间反复返工

## 7. 第一批实现顺序

第一批实现必须按以下顺序推进，不允许跳跃：

### 7.1 第 1 步：冻结底层契约

优先完成：

- 项目目录与服务拆分
- 统一接口规范
- 统一错误码
- 统一鉴权方案
- 核心表结构草案
- 事件 / 回调约定

原因：

- 这是后续所有模块的共同前提。

### 7.2 第 2 步：打通账号与身份

优先完成：

- 注册 / 登录
- 用户基础资料
- 买家 / 卖家身份选择
- 卖家认证申请与后台审核
- 买家自动认证规则

原因：

- 没有身份体系，商品、聊天、礼物权限无法落地。

### 7.3 第 3 步：打通商品主流程

优先完成：

- 商品发布
- 商品列表 / 详情
- 交易准则
- 下单前确认
- 订单创建

原因：

- 商品是平台核心交易入口。

### 7.4 第 4 步：打通支付与钱包

优先完成：

- 充值
- 支付
- 订单扣款
- 钱包流水
- 订单费率快照
- 礼物入账基础链路

原因：

- 所有交易动作必须先完成账务闭环。

### 7.5 第 5 步：打通 IM 与基础社交

优先完成：

- 好友 / 互关
- 会话列表
- 文本聊天
- 图片消息
- 基础审核接入

原因：

- 社交属性是平台差异化能力，但必须建立在账号与身份之上。

### 7.6 第 6 步：打通礼物与榜单

优先完成：

- 礼物列表
- 赠送礼物
- 收礼展示
- 基础榜单聚合

原因：

- 礼物与榜单依赖钱包、身份、用户主页数据。

### 7.7 第 7 步：打通审核与后台

优先完成：

- 商品审核
- 认证审核
- 提现审核
- 基础配置管理
- 审核日志与证据留存

原因：

- 后台是平台可运营的前提，但必须在主流程打通后接入。

## 8. 子 agent 分工建议

### 8.1 推荐拆分方式

建议按以下职责拆分子 agent：

- A 组：架构与契约组
- B 组：认证与用户组
- C 组：商品与订单组
- D 组：支付与钱包组
- E 组：IM 与社交组
- F 组：礼物与榜单组
- G 组：审核与后台组
- H 组：前端页面组
- I 组：测试与联调组

### 8.2 每组交付要求

每组都必须交付：

- 需求范围说明
- 接口定义
- 数据结构说明
- 状态流转说明
- 最小可运行代码
- 自测结果

## 9. 执行纪律

- 不得擅自扩展 MVP。
- 不得在未冻结契约前写实现。
- 不得绕过主服务直接改库。
- 不得在一个任务中同时修改多个核心状态机。
- 不得把“后续再补”写进当前版本的验收标准。
- 所有资金与审核动作必须保留可追溯记录。
- 所有跨服务改动必须先更新文档，再写代码，再联调。
- 所有新需求必须进入下一阶段，不得插入当前阶段。

## 10. 当前执行结论

当前项目的最优落地策略是：

1. 先冻结主闭环与服务边界。
2. 先实现身份、商品、订单、支付、钱包五个核心中枢。
3. 再补 IM、礼物、榜单与审核后台。
4. 最后再进入风控、地区权限、论坛和增长能力。

这套顺序的核心目标是：先让平台“可交易、可记账、可审核”，再让平台“可增长、可运营、可优化”。
