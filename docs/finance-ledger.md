# 财务账本协议草案（V0.1）

> 适用范围：二手闲置社交交易平台支付、钱包、礼物、商品、提现、退款、风控冻结、对账与清结算。
>
> 目标：冻结财务与支付领域的统一协议，作为后续子 agent、接口实现、数据库建模、风控与对账的共同约束。
>
> 核心原则：**所有资金动作必须账本化；禁止直接改余额；订单与费率必须快照；充值/消费/提现/退款必须有明确状态机；所有资金变更必须支持幂等、可追溯、可对账、可冻结、可解冻。**

## 1. 设计目标

1. 钱包余额与流水完全分离，余额只是账本汇总结果，不是业务真相来源。
2. 所有资金动作都必须生成标准流水，并可追溯到业务单据。
3. 商品订单、礼物订单、充值订单、提现订单、退款单都必须保留创建时的快照数据。
4. 后台费率修改只影响新单，不回溯历史单。
5. 支持冻结、解冻、待结算、可提现、退款冲正、人工补单、对账差异处理。
6. 首批实现优先保证：**充值、余额消费、礼物分账、提现、退款、账本、对账、幂等**。

## 2. 术语定义

- **钱包账户**：用户在平台内的财务账户视图，包含可用余额、冻结余额、可提现余额、待结算余额等。
- **流水**：一次不可变的资金变动记录。
- **账本**：由账户、流水、凭证、单据组成的资金事实系统。
- **业务单据**：订单、礼物单、提现单、退款单、风控单、人工调整单等。
- **费率快照**：单据创建或支付确认时固化的抽佣比例、分账比例、手续费、提现费等配置。
- **幂等键**：保证同一业务动作重复请求只会记一次账的唯一键。
- **冻结**：资金从可用状态转入冻结状态，不能消费或提现。
- **解冻**：冻结资金返回可用状态或按业务规则转入其他状态。
- **对账**：平台账本与支付渠道、退款渠道、提现渠道、人工调整记录之间的核对。

## 3. 总体财务原则

### 3.1 一切资金动作必须账本化

以下动作都必须写入流水：

- 充值入账
- 消费扣款
- 礼物扣款与分账
- 商品成交结算
- 提现冻结、提现扣减、提现失败回滚
- 退款冲正
- 风控冻结/解冻
- 人工补单/冲账
- 平台手续费入账
- 争议资金转移

### 3.2 不允许直接改余额

任何余额字段不得由业务代码直接写死更新，必须通过以下方式变更：

1. 创建资金流水
2. 写入资金分录/双分录明细
3. 由余额汇总器或事务内账本更新器统一计算余额

### 3.3 业务单据必须快照

单据生成时必须保存下列快照：

- 用户信息快照（用户 ID、身份类型、实名/认证状态）
- 商品快照（商品 ID、标题、价格、类目、交易准则）
- 礼物快照（礼物 ID、名称、单价、分账规则）
- 费率快照（平台服务费、礼物抽成、提现费、活动费率）
- 风控快照（当时的风控等级、冻结期限、审核结果）

快照必须随单据长期保留，不随后台配置变化而变化。

## 4. 钱包账户模型

### 4.1 账户粒度

建议采用“用户主钱包 + 子余额分层”的模型。

#### 账户主体

- `account_id`：钱包账户主键
- `user_id`：所属用户
- `account_type`：用户钱包 / 平台收入 / 平台准备金 / 退款准备金 / 风控冻结池
- `account_status`：正常 / 冻结 / 关闭
- `currency`：默认 CNY
- `version`：乐观锁版本号

#### 余额字段

建议至少保留以下余额视图：

- `available_balance`：可用余额，可消费、可转出、按规则可提现
- `frozen_balance`：冻结余额，暂不可消费/提现
- `withdrawable_balance`：可提现余额，满足提现条件的金额
- `pending_settlement_balance`：待结算余额，商品或礼物尚未进入可提现状态
- `refund_reserved_balance`：退款准备金，可选，用于争议期预留
- `total_balance`：总账面余额，原则上等于各分项之和

### 4.2 账户类型说明

#### 用户钱包账户

用户主钱包用于承接充值、消费、礼物收益、提现前资金归集。

#### 平台收入账户

用于承接平台服务费、礼物抽成、提现手续费等平台收入。

#### 风控冻结池

用于风控冻结场景下的资金暂存，不允许直接提现吗。

#### 退款准备金

用于退款、争议、失败订单回滚时的资金占位。

### 4.3 首批实现边界

第一批只实现以下账户能力：

- 用户钱包主账户
- 平台收入账户
- 冻结余额
- 可提现余额
- 待结算余额
- 退款冲正

不在第一批强制实现：

- 多币种
- 跨账户复杂清算网关
- 分级商户子账户
- 批量净额结算
- 复杂理财或利息逻辑

## 5. 流水模型

### 5.1 流水基本原则

每笔流水必须满足：

- 唯一
- 不可变
- 可追溯
- 可分类
- 可关联业务单据
- 可关联幂等键
- 可关联对账批次

### 5.2 流水建议字段

#### 基础字段

- `ledger_entry_id`：流水主键
- `ledger_no`：平台流水号，全局唯一
- `biz_type`：业务类型，如 recharge / consume / gift / withdraw / refund / freeze / unfreeze / adjustment
- `biz_id`：业务单据 ID
- `biz_no`：业务单号
- `user_id`：涉及用户
- `account_id`：账户 ID
- `direction`：in / out
- `amount`：金额，正数存储
- `balance_before`：变动前余额
- `balance_after`：变动后余额
- `status`：pending / succeeded / failed / reversed
- `source_channel`：微信 / 支付宝 / 余额 / 人工 / 系统
- `idempotency_key`：幂等键
- `ref_type`：关联对象类型
- `ref_id`：关联对象 ID
- `remark`：备注
- `occurred_at`：业务发生时间
- `created_at`：入账时间

#### 审计字段

- `request_trace_id`：请求追踪号
- `operator_type`：用户 / 系统 / 管理员 / 风控
- `operator_id`：操作人
- `snapshot_json`：关键快照
- `reconcile_batch_no`：对账批次号
- `reconcile_status`：未对账 / 已对账 / 差异 / 已调整

### 5.3 双分录要求

任何一笔资金动作，建议至少形成两个分录：

- 一方账户减少
- 另一方账户增加

例如：

- 用户充值：渠道待清算 -> 用户钱包可用余额
- 用户消费：用户钱包可用余额 -> 平台收入账户 / 卖家待结算账户
- 提现：可提现余额 -> 外部提现待支付 / 平台清算账户
- 退款：平台退款准备金 / 用户钱包 -> 原支付方向逆向回退

### 5.4 流水状态

- `pending`：待确认，常见于外部支付回调前
- `succeeded`：成功入账
- `failed`：失败，不影响余额
- `reversed`：已冲正

原则：**只有成功流水才能影响余额。**

## 6. 充值协议

### 6.1 充值目标

充值是平台内消费的统一入口，外部支付仅用于充值，不直接用于每次商品下单或送礼。

### 6.2 充值流程

1. 用户发起充值请求。
2. 系统创建充值订单，生成充值快照。
3. 固化充值金额、渠道、费率、活动信息。
4. 调起微信支付或支付宝支付。
5. 支付渠道回调验签成功后，充值订单转成功。
6. 生成充值入账流水。
7. 用户钱包可用余额增加。
8. 若满足认证条件，则触发买家自动认证规则。

### 6.3 充值状态机

- `created`：已创建
- `paying`：支付中
- `paid`：渠道支付成功
- `credited`：已入账
- `failed`：支付失败
- `closed`：超时关闭
- `refunded`：已退款

### 6.4 充值幂等

充值幂等键建议至少包含：

- `user_id`
- `recharge_order_no`
- `channel_trade_no`
- `pay_channel`

规则：同一充值订单的回调、补单、重试只能记一次账。

## 7. 扣款协议

### 7.1 扣款范围

扣款适用于：

- 商品购买
- 礼物赠送
- 订单保证金（如后续启用）
- 风控扣留转入冻结

### 7.2 扣款原则

- 先校验余额，再冻结或扣减。
- 扣款必须生成消费流水。
- 扣款金额必须使用订单快照中的金额，不可重新计算后覆盖历史。

### 7.3 扣款状态机

- `initiated`：已发起
- `reserved`：已冻结/预占
- `deducted`：已扣款
- `confirmed`：业务确认成功
- `reversed`：已冲正
- `failed`：扣款失败

### 7.4 扣款顺序建议

首批实现优先级：

1. 余额可用性校验
2. 创建扣款预留流水
3. 成功后确认扣减
4. 不成功则回滚预留

## 8. 礼物分账协议

### 8.1 礼物分账目标

礼物是平台社交激励和收入来源，必须同时满足：

- 买家扣款可追溯
- 平台抽成可追溯
- 收礼方收益可追溯
- 后续提现可追溯

### 8.2 礼物分账规则

礼物单创建时必须保存：

- 礼物单价
- 礼物数量
- 总金额
- 平台抽成比例
- 收礼方实得比例
- 活动补贴比例（如有）
- 对应榜单权重或魅力值规则

### 8.3 礼物资金流

推荐资金流：

`用户可用余额 -> 礼物消费扣款 -> 平台收入 + 收礼方收益待结算`

其中：

- 平台收入部分进入平台收入账户
- 收礼方收益进入待结算余额
- 若设置冻结期，则先进入冻结余额，等待解冻后再转可提现

### 8.4 礼物分账状态机

- `created`：礼物单已创建
- `paid`：已扣款
- `split_done`：已完成分账
- `settling`：待结算中
- `available`：可提现
- `frozen`：冻结中
- `cancelled`：取消
- `reversed`：冲正

### 8.5 风控规则

礼物分账可设置：

- 提现冷却期
- 单日收礼上限
- 高风险账号延迟到账
- 异常互刷冻结

## 9. 提现协议

### 9.1 提现目标

提现是用户将平台内可提现余额转出到外部支付渠道或银行卡的动作。

### 9.2 提现前置条件

- 账号通过实名或认证
- 提现金额满足最小阈值
- 满足冷却期要求
- 风控校验通过
- 不存在冻结资金占用

### 9.3 提现流程

1. 用户发起提现申请。
2. 系统校验可提现余额与风控状态。
3. 锁定提现金额，生成提现冻结流水。
4. 进入审核或自动处理队列。
5. 审核通过后发起外部打款。
6. 打款成功后转为提现成功流水。
7. 打款失败则解冻并回滚。

### 9.4 提现状态机

- `created`：已创建
- `reviewing`：审核中
- `frozen`：已冻结资金
- `processing`：打款处理中
- `paid`：打款成功
- `failed`：打款失败
- `rejected`：审核拒绝
- `cancelled`：用户取消
- `reversed`：已冲正

### 9.5 提现冻结与解冻规则

- 提现申请提交后，必须先冻结对应金额。
- 冻结期间不可再次消费或提现。
- 提现失败必须自动解冻。
- 人工驳回必须自动解冻。
- 平台风控冻结优先于提现放行。

## 10. 退款协议

### 10.1 退款原则

退款必须依据订单快照与原流水执行，不允许重新按当前商品或费率计算。

### 10.2 退款场景

- 商品未发货退款
- 交易争议退款
- 充值原路退款
- 风控误判退款
- 平台主动补偿退款

### 10.3 退款状态机

- `created`：已创建
- `reviewing`：审核中
- `approved`：已通过
- `processing`：退款处理中
- `refunded`：退款成功
- `failed`：退款失败
- `rejected`：已拒绝
- `reversed`：已冲正

### 10.4 退款顺序

优先按下列顺序处理：

1. 原支付渠道原路退回（优先）
2. 若原路不可行，再按平台内部账本冲正
3. 对已分账部分执行反向分录
4. 对平台收入部分按规则回退或保留

### 10.5 退款冲正规则

- 已结算给卖家的金额如需退款，应先冻结卖家可提现余额或从待结算中抵扣。
- 已提现成功的历史收益，若后续发生争议，必须记录追偿单，不可直接倒扣历史流水。

## 11. 费率快照协议

### 11.1 必须快照的费率

所有涉及金额计算的业务单据必须保存费率快照：

- 商品平台服务费比例
- 礼物平台抽成比例
- 收礼方分成比例
- 提现手续费比例或固定金额
- 退款手续费规则
- 活动补贴规则
- 特殊用户减免规则

### 11.2 快照规则

- 下单时快照，不允许事后按后台最新配置重算历史单。
- 订单详情页必须展示历史费率快照。
- 对账与审计必须使用同一快照字段。

### 11.3 费率变更原则

后台改费率后：

- 仅影响新建订单
- 旧单保持原快照
- 若存在未支付订单，是否重新应用费率必须显式定义，不默认变更

## 12. 幂等协议

### 12.1 幂等范围

以下接口必须幂等：

- 创建充值单
- 支付回调确认
- 消费扣款
- 礼物送出
- 提现申请
- 提现打款确认
- 退款申请
- 退款回调
- 冻结 / 解冻
- 人工补单

### 12.2 幂等键规范

幂等键建议由以下字段组合生成：

- `biz_type`
- `biz_no`
- `action_type`
- `user_id`
- `amount`
- `channel_trade_no`（如有）

### 12.3 幂等处理规则

- 同一幂等键重复请求必须返回同一结果。
- 已成功的资金动作不得再次入账。
- 处理中请求应返回已有单据状态。
- 失败请求可按业务重试，但必须换业务单号或保持同幂等逻辑。

## 13. 对账协议

### 13.1 对账目标

保证平台账本与外部渠道、业务单据、人工操作一致。

### 13.2 对账对象

- 微信支付渠道账单
- 支付宝渠道账单
- 提现打款渠道账单
- 退款渠道账单
- 平台内部账本流水
- 人工补单记录

### 13.3 对账维度

- 单笔对账
- 日对账
- 渠道对账批次
- 订单维度对账
- 用户维度对账
- 账户维度对账

### 13.4 对账结果分类

- 一致
- 平台多记
- 平台少记
- 渠道多记
- 渠道少记
- 金额不一致
- 状态不一致
- 单号不匹配

### 13.5 差异处理原则

- 差异必须进入差异单，不允许口头修正。
- 差异修正必须生成调整流水。
- 修正后保留原始差异证据。
- 对账修正必须可审计、可回放。

## 14. 冻结与解冻规则

### 14.1 冻结类型

- 提现冻结
- 风控冻结
- 退款冻结
- 争议冻结
- 管理员手工冻结

### 14.2 冻结原则

- 冻结只能减少可用余额，不得凭空增加余额。
- 冻结必须生成冻结流水。
- 冻结金额必须与业务单据绑定。
- 冻结原因必须记录。

### 14.3 解冻原则

- 解冻必须有明确触发条件。
- 解冻必须生成解冻流水。
- 解冻后资金回到可用余额、待结算余额或可提现余额，不能丢失归属。

### 14.4 风控冻结

当出现以下情况时可冻结：

- 异常频繁提现
- 互刷礼物
- 订单争议
- 账号异常登录
- 设备/IP 风险
- 疑似洗钱或套利

## 15. 首批实现边界

### 15.1 第一批必须实现

1. 钱包账户基础表
2. 资金流水表
3. 充值订单与回调入账
4. 余额消费扣款
5. 礼物送出与分账
6. 提现申请、冻结、打款、失败回滚
7. 退款申请与冲正
8. 费率快照
9. 幂等控制
10. 日对账与差异单
11. 冻结/解冻
12. 资金审计日志

### 15.2 第一批可延期

- 多渠道复杂聚合清算
- 自动化分账引擎的高级规则编排
- 跨境/多币种
- 税务发票
- 复杂营销补贴引擎
- 复杂账户体系的多级商户结算

### 15.3 第一批不允许做的事

- 直接更新余额字段绕过流水
- 不保存订单快照
- 不保存费率快照
- 不记录幂等键
- 退款不冲正原流水
- 提现不冻结先打款
- 对账差异直接人工改余额

## 17. 推荐实现约束

### 17.1 事务边界

以下动作必须在同一数据库事务内完成：

- 校验余额
- 写流水
- 更新账户汇总余额
- 写业务单状态

### 17.2 并发控制

- 钱包账户更新必须有乐观锁或行锁保护。
- 同一账户并发扣款必须避免超扣。
- 幂等键必须唯一索引。

### 17.3 可观测性

所有资金动作必须记录：

- 请求 ID
- 业务单号
- 幂等键
- 操作人
- 渠道单号
- 变动前后余额
- 对账状态
- 风控状态

## 18. 建议的统一资金流约定

### 18.1 充值

`外部支付渠道 -> 平台充值订单 -> 用户钱包可用余额`

### 18.2 商品消费

`用户钱包可用余额 -> 平台收入 / 卖家待结算`

### 18.3 礼物消费

`用户钱包可用余额 -> 平台收入 / 收礼方待结算`

### 18.4 提现

`可提现余额 -> 冻结 -> 外部打款成功/失败回滚`

### 18.5 退款

`按订单快照 -> 原路退款或账本冲正`

## 19. 验收标准

第一批实现完成后，至少满足：

1. 任意资金变动都能查到流水。
2. 任意流水都能追溯到业务单据。
3. 任意订单都能查到费率快照。
4. 充值、消费、提现、退款均可幂等。
5. 冻结金额不会被重复消费或提现。
6. 对账差异可以定位到具体单笔流水。
7. 历史订单不受后台费率修改影响。
8. 不存在“直接改余额”的业务路径。

## 20. 结论

本项目财务模块的第一原则不是“余额显示正确”，而是“**账本事实正确**”。

后续所有支付、钱包、礼物、提现、退款、对账、风控子模块，必须以本草案作为统一约束：

- 先单据，后流水
- 先快照，后结算
- 先冻结，后放行
- 先幂等，后入账
- 先对账，后修正
- 只记账，不直改
